    Javascript Components

    First up, Components. Component is a package manager for CommonJS style modules for browsers. Most of the code I write these days is in this style. It helps separate functionality into discrete, testable modules.

    Hyperbone

    Strictly speaking this is on a different Github account, but for the sake of completeness, Hyperbone is a Hypermedia Framework built on Backbone with a significantly better built in view engine for binding models directly to the DOM (it is more like Angular in that respect).

    In a way the fancy evented models and view-model binding is sort of a bonus. The real power comes in terms of defining 'commands' to interact with a web service. This commands can come directly from a Hypermedia API (REST Level 3) or they can be defined as defaults for your models in the client to define server side interactions in advance of the server side being developed or to consume a non-RESTful or non Hypermedia API without needing to change the client side design pattern.

    It's still in heavy active development and some of the implications of the focus on Hypermedia are only just becoming apparent to me. The TODOMVC app has been implemented with Hyperbone here with a live demo here.

    I'd argue that this isn't quite ready for public use yet but it's getting there.

    Tween

    Tween morphs values from start to end over time. You can even morph '#FF0000' to '#F8E423' if you like. It has a fluent interface and makes use of other components I've written to perform easing on the morphs using bezier curves. It can even emulate CSS3 easing.

    var tween = require('tween');
var animation = tween({ x : 10, y : 10 })
  .to({ x : 100, y : 200 })
  .with('ease-in')
  .duration('5s')
  .tick(function( tweenObject ){
     // note computeMatrix3D doesn't come with Tween.
     myElement.style['-webkit-transform'] = computeMatrix3D( tweenObject );
  })
  .play()

    I've used this module for animation and creating gain envelopes for Web Audio. It's useful whenever you need to change any value over time with a specific easing curve.

    Easing

    Normalised Cubic-bezier based easing generator. Given a particular easing curve it computes a value between 0 and 1 for a given time (also a number between 0 and 1).

    It has presets, a css3 easing emulation (where the eased value is generated by moving along the x axis of the curve at a linear speed and reading where the y axis intersects) and the patently useless Einstein mode (where the eased value is generated by moving along the curve at a linear speed and reading where the y axis intersects which tends to produce much more subtle effects and bizarre effects.

    The main difference is that an Einstein mode custom curve can be any arbitrary curve - it does not have to start at 0,0 and end at 1.1. It probably should though.

    It has a fluent interface, but it's a little clumsy to my eyes these days.

    // presets
var myEaser = require('easing').Ease().using("ease-in");
myEaser(0.5);
// 0.3147198334560001

// css3 emulation
var myEaser = require('easing').Ease().usingCSS3Curve(0.33, -0.305, 0.715, -0.155);
myEaser(0.5);
// -0.06145341884495001

// Einstein mode
myEaser = require('easing').Ease().usingCustomCurve({c1 : [0,0], c2: [0.075, 0.61], c3: [0.36, 0.93], c4: [1,1]})
// 0.7025

    Tick

    Register callbacks to be run on browser animation frames. Do a global pause and resume. Pause, resume and stop individual handlers, too. This is surprisingly handy.

    var tick = require('tick');

var handle = tick.add(function(elapsed, stop){
  if (elapsed > 5000){
    stop(); // this callback won't fire again
  }
});

handle.pause();
tick.pause();
tick.resume();

    Tick
    Covered by tests? No. This was tested manually as the test framework I was using at the time wasn't good enough.

    Bezier

    I've been refactoring bezier curve functions for years now. This was the last (and, I think, final) attempt. It supports linear, cubic and quadratic curves and supports point data (getting the value of x,y at a given % along the curve) - which can come in css, object or array flavours - and finding the value of Y at X, which is based on building a lookup table with a definable number of samples.

    var curve = require('bezier')();
curve
  .c1([10,15])
  .c2([20,25])
  .c3([30,35])
  .c4([40,45])
curve.point(0.25)
// {x : 11.5625, y : 20.625 }
curve
  .buildLookup()
  .findYAtX(0.5)

    Covered by tests? Yes

    Simple module to return the browser specific CSS strings for Transition and Timing functions or false if they're not supported, useful if you're writing an animation library that wants to do animations using native CSS where supported.

require('detect-transition').timingFunction

    Parse Duration

    Turns human friendly duration strings into computer friendly counts of milliseconds. Supports ms, s, m, h, d and w. 

    var parse = require('parse-duration');
parse(1000);
> 1000
parse('100ms')
> 100
parse('100s')
> 6000000..

    Gather

    Batch processing for asynchronous tasks. Get a callback for when a bunch of other async callbacks are done. This module is a bit of an old workhorse really. One of the first modules I wrote where I really cared passionately about the API.

    var gathering = require('gather').gathering();

gathering.task(function(done, error){
  setTimeout(function(){
    // do some work..
    done();
  }, 1000)
});
gathering.task(function(done, error){
  setTimeout(function(){
    // do some different work.
    done();
  }, 500)
});
gathering.update(function( percent ){
  // every time a task is 'done' I get the percent
  // done here!
})
gathering.run(function(errors){
  //errors is an array of any errors, otherwise they're all done now.
}, 2000) // add a time limit

    Real Time Interactive JS IDE

    This was a hilarious gag for a Developer Led training session. I hacked a pseudo-real time web based IDE together (using ACE), so that I could demonstrate various features of Javascript to other developers without switching between source code and the output. Everything was in one page. As I changed things in the source, it would update the output immediately. Just to make sure it was complete overkill changes to the source were transmitted to the server over socket.io as diff patches rather than sending the whole file. Just because.

    Far too afraid to look at the source for this. It's from my pre-module days when everything was about extending objects and object prototypes, data was just data and everything was co-ordinated using a pub/sub module.

    Hopefully this is soon to be redundant, but it transforms the build output from Component to a simple flat single level structure doing away with the need for require.alias calls, reducing the file size before and after minification. Totally unnecessary if you've got gzip on your server, but may speed up JS parsing anyway.

    Dusty Old Stuff

    This is JS I wrote before I started writing in modules. They're listed here more for historical reasons.

    Planet

    When I was writing the UI for Pathways I needed to write a tiny, minimalist Vector graphics library that would output VML, SVG or Canvas from a standardised input. I found various CSS3 polyfills to be unacceptable in terms of performance, and other Vector graphics libs too over featured and bloated.

    Planet was... well, tiny but useful. It only supported a few vector primitives - boxes with rounded edges, lines etc.

    This was written when I was still doing 'object style' JS, inspired by the way jQuery is programmed. Planet is built by extending the Planet object and the Planet's prototype object, and instead of a cool fluent interface it just takes a giant uncomprehensible splurge of JSON data.

    I've learnt a lot since writing this.

    Planet
    Covered by tests: No. How to you test Canvas output?

    Cassette-Express

    One of my clients used Cassette (a .net nuget package) quite extensively to manage their Javascript assets. We used it to assemble random bits of javascript into bundles, sort of a naive old-school way of breaking large Javascript applications into separate files. It especially suits the Object style of Javascript programming where you have one global variable, your namespace, and each file extends that object in some way, and your third party dependencies are just random javascript files you want to insert in the page.

    Cassette-Express was a port of this to node, with the idea that instead of assembling bundles as a build step, it can be done at run time instead.

    I spent absolutely ages writing code to read and resolve dependencies to get this working, and it was, I think, a total triumph. Except almost immediately after finishing it I moved over to using Component as a package manager and writing in module style instead.

    It possibly still has some real world use for people who absolutely refuse to write the JS as modules and use Component or Browserify.

    Cassette-Express
    Covered by tests: Yes
